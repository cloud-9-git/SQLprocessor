# SQLprocessor Study Guide

## 이 문서의 목적

발표 전에 "코드가 왜 이렇게 동작하는지"를 빠르게 이해하기 위한 요약 노트입니다.  
전체 코드를 외우는 대신, 입력이 실행 결과로 바뀌는 흐름만 잡으면 됩니다.

## 1. 먼저 이해해야 할 큰 흐름

```text
SQL 파일 입력
-> main.c가 문장 단위로 분리
-> lexer.c가 토큰화
-> parser.c가 Statement 생성
-> executor.c가 실행
-> CSV 파일에 반영
```

## 2. 파일별 역할

| 파일 | 꼭 이해할 내용 |
| --- | --- |
| `main.c` | 파일 경로를 받고 `fgetc`로 문자를 읽으며 SQL 문장을 나눔 |
| `lexer.c` | `SELECT`, `INSERT`, 문자열, 숫자 등을 토큰으로 구분 |
| `parser.c` | SQL 문장을 `Statement` 구조체에 담음 |
| `executor.c` | 조건 검사 후 실제 조회/삽입/수정/삭제 수행 |
| `types.h` | 전체 프로그램이 공유하는 데이터 구조 정의 |

## 3. 지금 코드 기준으로 꼭 알아둘 점

| 주제 | 설명 |
| --- | --- |
| 컴파일 방식 | `main.c` 맨 아래에서 `lexer.c`, `parser.c`, `executor.c`를 직접 포함하므로 `gcc -g main.c -o sqlsprocessor`처럼 `main.c`만 컴파일해야 함 |
| 입력 처리 | 파일을 한 줄이 아니라 문자 단위로 읽어서 주석, 따옴표, 세미콜론을 정확히 구분함 |
| 실행 지원 | 현재 코드는 `INSERT`, `SELECT`뿐 아니라 `UPDATE`, `DELETE`도 실행함 |
| 저장 방식 | 변경은 메모리 캐시에 반영한 뒤 CSV 파일 전체를 다시 씀 |
| 제약조건 | `INSERT`와 `UPDATE`에서 `PK`, `UK`, `NN`을 검사함 |

## 4. 발표에서 설명해야 하는 핵심 로직

### 4-1. 왜 `main.c`에서 바로 실행하지 않고 파싱 단계를 두는가?

SQL 문자열을 그대로 실행하면 구조를 알 수 없습니다.  
그래서 먼저 `Statement`로 바꿔서 "이 문장이 SELECT인지, 어느 테이블인지, WHERE 조건은 무엇인지"를 분리합니다.

### 4-2. 왜 `;`를 기준으로 문장을 자르는가?

한 파일 안에 여러 SQL 문장이 들어갈 수 있기 때문입니다.  
단, 문자열 안의 `;`는 문장 끝이 아니므로 따옴표 상태를 같이 추적합니다.

### 4-3. 왜 `main.c`가 `fgetc`로 한 글자씩 읽는가?

줄바꿈이 섞여 있어도 문제없고, 한 줄에 SQL 여러 개가 있어도 처리할 수 있기 때문입니다.  
또 `--` 주석인지, 따옴표 안의 문자 인지, 세미콜론이 문장 끝인지도 문자 단위로 더 정확히 구분할 수 있습니다.

### 4-4. 왜 `parse_csv_row`에서 따옴표를 신경 쓰는가?

`'tony,stark@test.com'`처럼 값 안에 쉼표가 있으면 일반적인 `split(',')`은 깨집니다.  
그래서 따옴표 안에 있는 쉼표는 컬럼 구분자로 처리하지 않습니다.

### 4-5. 왜 PK는 따로 인덱스를 유지하는가?

PK 중복 여부를 빠르게 확인하기 위해서입니다.  
정렬된 `pk_index` 배열에 저장해 두고, 중복 검사는 이분 탐색으로 수행합니다.

### 4-6. 왜 파일을 일부만 수정하지 않고 전체를 다시 쓰는가?

구현을 단순하고 안전하게 유지하기 위해서입니다.  
메모리 캐시에 현재 상태를 유지한 뒤, 변경이 끝나면 파일 전체를 다시 저장합니다.

## 5. 제약조건 이해하기

| 제약 | 의미 | 예시 |
| --- | --- | --- |
| `PK` | 기본키, 중복 불가 | 같은 `id` 두 번 삽입 불가 |
| `UK` | 고유값, 중복 불가 | 같은 `email` 두 번 삽입 불가 |
| `NN` | 빈 값 불가 | `pwd`를 `''`로 저장 불가 |

## 6. 테스트를 볼 때 체크할 것

| 테스트 파일 | 무엇을 증명하는가 |
| --- | --- |
| `demo_select.sql` | 전체 조회와 조건 조회가 되는가 |
| `demo_insert.sql` | 새 데이터가 실제로 저장되는가 |
| `demo_insert_error.sql` | PK, UK 중복이 차단되는가 |
| `demo_edge_case.sql` | 조회 결과 없음, quoted 값, 제약 위반, 대상 없는 수정/삭제 같은 예외를 막는가 |
| `demo_update.sql` / `demo_delete.sql` | 수정과 삭제가 정상 반영되는가 |

## 7. 현재 시연 흐름을 코드와 연결해 보면

1. `demo_reset.sql`로 기준 상태를 맞춥니다.
2. `demo_select.sql`로 조회 흐름을 보여줍니다.
3. `demo_insert.sql`로 정상 INSERT를 보여줍니다.
4. `demo_insert_error.sql`로 PK와 UK 중복을 보여줍니다.
5. `demo_edge_case.sql`로 예외 상황을 묶어서 보여줍니다.
6. `demo_update.sql`, `demo_delete.sql`로 CRUD를 마무리합니다.

## 8. 단위 테스트 관점에서 보면

| 함수 | 최소 검증 포인트 |
| --- | --- |
| `get_next_token` | 키워드, 숫자, 문자열을 올바르게 토큰화하는가 |
| `parse_statement` | SELECT / INSERT 문장을 정확히 구조화하는가 |
| `parse_csv_row` | 따옴표 안 쉼표를 컬럼 구분자로 오인하지 않는가 |
| `compare_value` | 공백과 따옴표 유무가 달라도 같은 값으로 비교되는가 |
| `execute_insert` | PK/UK/NN 위반 시 삽입을 막는가 |
| `execute_update` | PK 변경 금지, UK/NN 위반 차단이 되는가 |
| `execute_delete` | 조건에 맞는 행만 삭제하고 없으면 알림을 주는가 |

## 9. 예상 질문과 짧은 답변

### Q. 왜 DBMS 대신 CSV를 사용했나요?

과제의 핵심은 저장 엔진 자체보다 `SQL -> 파싱 -> 실행` 흐름을 구현하는 것이기 때문입니다.  
CSV는 저장 구조가 단순해서 처리기 동작을 설명하기 좋습니다.

### Q. 최소 요구사항은 INSERT, SELECT인데 왜 UPDATE, DELETE도 넣었나요?

실제 CRUD 전체 흐름을 보여주면 처리기 구조를 더 명확히 설명할 수 있기 때문입니다.  
추가 구현 요소로도 차별점이 됩니다.

### Q. 가장 중요한 함수는 무엇인가요?

`main`, `parse_statement`, `execute_insert`, `rewrite_file`입니다.  
이 네 부분만 이해해도 입력부터 저장까지 전체 흐름을 설명할 수 있습니다.

### Q. 가장 까다로운 엣지 케이스는 무엇인가요?

쉼표나 세미콜론이 들어간 quoted 값, 중복 PK/UK, 빈 NN 값, 대상이 없는 UPDATE/DELETE입니다.

## 10. 발표 직전 1분 복습

1. 입력은 SQL 파일이다.
2. 파싱 결과는 `Statement` 구조체다.
3. `main.c`는 `fgetc`로 주석, 따옴표, 세미콜론을 구분한다.
4. 실행은 `executor.c`가 담당하고, 저장은 CSV 전체 재기록 방식이다.
5. 테스트는 정상 흐름, 중복 제약, 엣지 케이스를 모두 보여준다.
